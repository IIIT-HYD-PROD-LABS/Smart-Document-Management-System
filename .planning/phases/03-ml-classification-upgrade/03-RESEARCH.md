# Phase 3: ML Classification Upgrade - Research

**Researched:** 2026-03-10
**Domain:** scikit-learn document classification, TF-IDF text classification, evaluation metrics, Next.js frontend
**Confidence:** HIGH

## Summary

Phase 3 upgrades the existing document classifier from 76.4% to >85% accuracy on real data. The codebase already has a complete ML pipeline: dataset download (7 Kaggle sources), data preparation with OCR, TF-IDF + Logistic Regression/Naive Bayes training with GridSearchCV, and evaluation report generation. The primary bottleneck is data volume -- the initial training used only `--max-per-category 50` (1,210 total samples). Increasing to 500-1000+ per category should yield the accuracy jump based on the learning curve pattern (100% on categories with good data coverage like UPI/tickets, 57-73% on undercovered ones like bills/invoices).

The training pipeline (`train.py`) already generates confusion matrices, classification reports, and per-category precision/recall/F1. The evaluation report is saved as JSON at `backend/models/evaluation/evaluation_report.json`. What's missing: (1) a backend API endpoint to serve evaluation metrics, (2) frontend confidence badge with color-coded indicators (green/yellow/red), and (3) a frontend model evaluation report view.

**Primary recommendation:** Increase training data volume via existing pipeline (`--max-per-category 1000`), tune TF-IDF/model hyperparameters if needed, add API endpoint for evaluation report, and implement color-coded confidence badges in the document list UI.

### Anti-Patterns to Avoid
- **Training on synthetic data when real data is available:** The synthetic data was a bootstrap; real data is now available. Use `--real-only` or `--combined` mode, never `--synthetic-only` for final model.
- **Overfitting to test set:** The existing code properly splits into train/val/test. Do NOT tune hyperparameters on test set. Use validation set for model selection, test set for final reporting only.
- **Increasing max_features beyond vocabulary:** The initial training had 4,392 vocab with max_features=5000. With more data, vocabulary will grow. Increase max_features proportionally (e.g., 10000-15000).
- **Mixing confidence threshold logic with classification:** The model's `ML_CONFIDENCE_THRESHOLD` (0.3) in config.py is for "unknown" fallback. The UI color thresholds (80%/50%) are a separate concern.

## Common Pitfalls

### Pitfall 1: Class Imbalance in Training Data
**What goes wrong:** With `--max-per-category 1000`, some categories (UPI from CSV: thousands of rows) will hit the cap while others (tickets: limited image sources) may have far fewer. This creates imbalanced training.
**Why it happens:** Different dataset sources have vastly different sizes. CSV datasets (UPI, bank) are large; image datasets require OCR and have variable quality.
**How to avoid:** Use stratified sampling (already done in train_test_split with `stratify=`). Consider adjusting `class_weight='balanced'` in LogisticRegression. Monitor per-category recall in eval report.
**Warning signs:** High overall accuracy but specific categories below 70%.

### Pitfall 2: OCR Quality Degrading Training Data
**What goes wrong:** Low-quality OCR from scanned documents produces garbage text that pollutes training data.
**Why it happens:** Some dataset images are low resolution, skewed, or multi-column.
**How to avoid:** The `min_text_length=20` filter in prepare.py helps. Consider raising it to 50. Review samples from low-performing categories.
**Warning signs:** Categories with mostly image-sourced data (invoices, bills) performing worse than CSV-sourced (UPI, bank).

### Pitfall 3: TF-IDF Vocabulary Mismatch Between Training and Production
**What goes wrong:** Training data vocabulary (financial document keywords) doesn't match production documents well.
**Why it happens:** Training data from Kaggle may have different formatting/language than actual user uploads.
**How to avoid:** Use `--combined` mode to blend real datasets with synthetic data. The synthetic data in train.py contains Indian financial terminology that bridges the gap.
**Warning signs:** Good test accuracy but poor real-world performance on user uploads.

### Pitfall 4: Data Leakage from Augmentation
**What goes wrong:** Augmented versions of the same document appear in both train and test sets.
**Why it happens:** If synthetic augmentation happens before train/test split.
**How to avoid:** The current code generates augmented data and splits after. This is fine for synthetic. For real data, augmentation should happen AFTER splitting. The current code does NOT augment real data (only synthetic), so this is safe.
**Warning signs:** Test accuracy much higher than real-world accuracy.

### Pitfall 5: Frontend Color Thresholds Not Matching Requirements
**What goes wrong:** Using wrong thresholds or wrong direction (0-1 vs 0-100).
**Why it happens:** Backend returns confidence as 0.0-1.0 float; UI might display as percentage.
**How to avoid:** Requirements specify: green >80%, yellow 50-80%, red <50%. Backend stores as 0.0-1.0. Compare against 0.8 and 0.5, not 80 and 50.
**Warning signs:** All badges showing same color.

### Confidence Badge Component (frontend)
```tsx
// Color-coded confidence badge per AIML-03 requirements
function ConfidenceBadge({ score }: { score: number }) {
  const pct = Math.round(score * 100);
  let colorClass: string;

  if (score >= 0.8) {
    colorClass = "bg-[#10b981]/10 text-[#10b981]";  // Green
  } else if (score >= 0.5) {
    colorClass = "bg-[#f59e0b]/10 text-[#f59e0b]";  // Yellow
  } else {
    colorClass = "bg-[#ef4444]/10 text-[#ef4444]";  // Red
  }

  return (
    <span className={`text-[11px] px-2 py-0.5 rounded ${colorClass}`}>
      {pct}%
    </span>
  );
}
```

## Open Questions

1. **OCR processing time for 1000+ images per category**
   - What we know: Previous run with 50/category was fast. 1000/category means ~6000 OCR operations.
   - What's unclear: How long OCR will take in Docker container for 6000+ images. Could be 30-60+ minutes.
   - Recommendation: Run preparation with `--skip-ocr` first to verify CSV data, then do full run. Consider showing progress.

2. **Whether LinearSVC actually improves over LR for this dataset**
   - What we know: For TF-IDF text classification, SVM and LR typically perform similarly. LR already does well.
   - What's unclear: With more data, which will be better.
   - Recommendation: Include SVM as option but don't depend on it. More data is the primary lever.

3. **Tickets category data source**
   - What we know: Tickets got 100% accuracy with only 50 samples. No dedicated ticket dataset in the 7 Kaggle sources.
   - What's unclear: Whether ticket data comes from image datasets or only from synthetic. If mostly synthetic, real-world performance may be worse.
   - Recommendation: Verify data sources for tickets category in the prepared CSV. May need to rely on synthetic augmentation for tickets.
