**3. Configure backend/alembic/env.py:**

Replace the auto-generated env.py with a custom version that imports the app's models and settings. The complete file:

```python
"""Alembic migration environment configuration."""
from logging.config import fileConfig
from sqlalchemy import engine_from_config, pool
from alembic import context

# Import app settings and Base
from app.config import settings
from app.database import Base

# CRITICAL: Import ALL models so they register with Base.metadata.
# If a model is not imported here, Alembic autogenerate will generate
# a DROP TABLE migration for that model's table.
from app.models.user import User          # noqa: F401
from app.models.document import Document  # noqa: F401
from app.models.refresh_token import RefreshToken  # noqa: F401

config = context.config

# Override sqlalchemy.url from alembic.ini with the app's DATABASE_URL
config.set_main_option("sqlalchemy.url", settings.DATABASE_URL)

if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata


def run_migrations_offline() -> None:
    """Run migrations in 'offline' mode (generate SQL without connecting)."""
    url = config.get_main_option("sqlalchemy.url")
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )
    with context.begin_transaction():
        context.run_migrations()


def run_migrations_online() -> None:
    """Run migrations in 'online' mode (connect to database and execute)."""
    connectable = engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
        )
        with context.begin_transaction():
            context.run_migrations()


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

Key details:
- All three model files are imported with `# noqa: F401` to suppress unused import warnings.
- `settings.DATABASE_URL` overrides the alembic.ini URL so there's one source of truth.
- Both offline and online modes are supported.

**1. Generate the initial migration:**

Run from the backend directory:
```bash
cd backend
alembic revision --autogenerate -m "initial schema - users, documents, refresh_tokens"
```

This will generate a migration file in `backend/alembic/versions/` that contains:
- `upgrade()`: CREATE TABLE for users, documents, refresh_tokens with all columns, indexes, and foreign keys.
- `downgrade()`: DROP TABLE for all three tables.

**Verify the generated migration:**
- Open the generated file and confirm it has CREATE TABLE for `users`, `documents`, and `refresh_tokens`.
- Confirm it does NOT have any DROP TABLE operations in `upgrade()`.
- Confirm indexes are included (e.g., `idx_documents_category_user`, `idx_refresh_tokens_user_id`).

If the autogenerate produces unexpected results (DROP TABLE, missing tables), check that all models are imported in env.py.

**IMPORTANT NOTE for execution:**
- If a database already exists with tables from `create_all()`, autogenerate may produce an empty migration (because the schema matches). In that case, the migration file will have empty `upgrade()` and `downgrade()` functions. This is correct -- it means the database is already at the target state.
- For an existing database: after generating the migration, run `alembic stamp head` to mark it as current WITHOUT executing any SQL.
- For a fresh database: run `alembic upgrade head` to create all tables via the migration.
- Add a comment at the top of the migration file explaining this distinction.
